Overview

Gitの基本的な流れ

コマンドを覚える前に、データがどこからどこへ動くか把握しておきましょう。

あなたのPC
作業ディレクトリ

→
一時保管場所
ステージング
git commit
→
ローカル
リポジトリ
git push
→
クラウド

💡
かんたんなたとえ
「作業ディレクトリ」は机の上、「ステージング」は封筒に入れる前の書類、「コミット」は封筒に封をして棚に保管、「プッシュ」はその棚をクラウドにコピーするイメージです。
📝
基本操作
毎日必ず使うコマンド群。まずここをマスターしよう。
🆕 こんなとき：新しいプロジェクトを始めるとき
リポジトリを新規作成する
使うシーン：まだ Git 管理されていないフォルダを、初めて Git リポジトリとして設定したいとき。このコマンドを実行すると .git フォルダが作られます。
> git init
💡
実行するとフォルダが Git リポジトリになります（.git フォルダが作られます）
📥 こんなとき：GitHubのリポジトリを手元にコピーしたいとき
リポジトリをクローンする
使うシーン：チームのリポジトリや、他人の GitHub プロジェクトを自分のパソコンにダウンロードするとき。URL は GitHub のリポジトリページの「Code」ボタンからコピーできます。
基本（フォルダ名はリポジトリ名になる）
> git clone https://github.com/ユーザー名/リポジトリ名.git
フォルダ名を指定してクローンしたい場合
> git clone https://github.com/ユーザー名/リポジトリ名.git my-folder
✅ こんなとき：変更したファイルをコミット対象に入れたいとき
ファイルをステージに追加する（add）
使うシーン：編集したファイルを「次のコミットに含める」と印を付ける操作です。複数ファイルを変更しても、add したものだけがコミットされます。
特定のファイルだけ追加（おすすめ）
> git add index.html
変更したファイルをすべて追加
> git add .
特定フォルダ内のファイルをすべて追加
> git add pages/
💾 こんなとき：作業の区切りで変更を記録したいとき
変更をコミット（記録）する
使うシーン：ステージに上げたファイルを正式に記録します。メッセージには「何をしたか」を日本語でOKなので簡潔に書きましょう。
基本（add したファイルをコミット）
> git commit -m "ナビゲーションのリンクを修正"
add と commit を同時にやる（追跡済みファイルのみ）
> git commit -am "スタイルを調整"
💡
コミットメッセージのコツ：「何を」「なぜ」を書くと後から見返したときに分かりやすい。例: 「ヘッダー画像を追加」「ログインエラーを修正」
🌿
ブランチ操作
作業を分岐させてチーム開発をスムーズに。
🌿 こんなとき：今どのブランチにいるか確認したいとき
ブランチ一覧を見る
使うシーン：今いるブランチを確認したり、存在するブランチをリストアップしたいとき。* が付いているのが現在のブランチです。
ローカルのブランチ一覧
> git branch
リモートも含めた全ブランチ一覧
> git branch -a
✨ こんなとき：新しい機能や修正の作業を始めるとき
新しいブランチを作って移動する
使うシーン：新機能の追加やバグ修正をするとき、main ブランチを直接変えないように専用のブランチを作って作業します。
新しいブランチを作成して、そのブランチに移動（従来のコマンド）
> git checkout -b feature/ログイン機能
最新の Git では switch コマンドが推奨
> git switch -c feature/ログイン機能
🔀 こんなとき：別のブランチに切り替えたいとき
既存のブランチに移動する
使うシーン：作業中のブランチから別のブランチへ移りたいとき。移動する前にコミットかスタッシュをしておきましょう。
main ブランチに移動
> git checkout main
別のブランチに移動
> git checkout feature/ログイン機能
最新の Git では switch コマンドが推奨
> git switch main
🤝 こんなとき：作業ブランチの内容をmainに取り込みたいとき
ブランチをマージ（合流）する
使うシーン：作業ブランチでの変更を main ブランチに取り込むとき。GitHub では Pull Request を使う方法が一般的です。
手順1：まず main ブランチに移動する
> git checkout main
手順2：作業ブランチを main にマージする
> git merge feature/ログイン機能
🗑️ こんなとき：マージ済みの不要なブランチを片付けたいとき
ブランチを削除する
使うシーン：Pull Request がマージされた後、不要になった作業ブランチを削除してリポジトリを整理します。
マージ済みブランチを削除（安全）
> git branch -d feature/ログイン機能
未マージでも強制削除（注意して使う）
> git branch -D feature/不要なブランチ
⚠️
-D（大文字）は未マージのコミットも含めて強制削除します。作業中の内容が消えるので慎重に！
☁️
リモート（GitHub）との同期
ローカルとGitHubのリポジトリを同期させるコマンド。
📤 こんなとき：ローカルの変更をGitHubにアップしたいとき
変更をプッシュ（アップロード）する
使うシーン：コミットした変更を GitHub に反映させたいとき。初回は -u オプションをつけてリモートとの追跡関係を設定します。
初回プッシュ（-u でリモートとの追跡設定も同時に）
> git push -u origin ブランチ名
2回目以降（追跡設定済みなら省略形でOK）
> git push
📥 こんなとき：チームの最新変更を取り込みたいとき
最新の変更をプル（ダウンロード）する
使うシーン：他のメンバーが GitHub にプッシュした変更を自分のパソコンに取り込むとき。作業前に毎回 pull する習慣をつけると競合が減ります。
基本（現在のブランチを最新に更新）
> git pull
特定のリモートとブランチを指定する場合
> git pull origin main
💡
毎朝最初にやること：git pull で最新状態にしてから作業を始めると、コンフリクト（競合）が起きにくくなります。
🔍 こんなとき：GitHubの状態を確認だけしたいとき
リモートの情報を取得する（マージはしない）
使うシーン：pull は取得＋マージをまとめて行いますが、fetch は取得だけで自分の作業には影響しません。「まず確認だけしたい」ときに使います。
手順1：リモートの情報を取得する（作業には影響なし）
> git fetch
手順2：取得後に自分の変更との差分を確認する
> git diff HEAD origin/main
🔎
確認・調査コマンド
今の状態・変更内容・履歴を調べる。
👀 こんなとき：どのファイルが変更されているか確認したいとき
現在の状態を確認する
使うシーン：何か操作する前に「今どういう状態か」を確認するために使います。迷ったらまずこれを実行！
> git status
📖
出力の見方：
On branch main ← 今いるブランチ
Changes to be committed: ← add 済み（コミットされる）
Changes not staged: ← 変更あるが add していない
Untracked files: ← Git 管理外の新規ファイル
📜 こんなとき：コミット履歴を見たいとき
コミット履歴を表示する
使うシーン：過去にどんな変更をしたか振り返りたいとき、または特定のコミットIDを調べたいとき。
詳細表示
> git log
1行で見やすく表示（おすすめ）
> git log --oneline
ブランチの分岐もグラフで表示
> git log --oneline --graph --all
🔍 こんなとき：何を変更したか詳しく見たいとき
変更内容の差分を表示する
使うシーン：コミット前に「何を変えたか」確認したり、ブランチ間の差分を比較したいとき。
ステージ前（add していない）の変更を確認
> git diff
ステージ済み（add 後）の変更を確認
> git diff --staged
ブランチ間の差分を確認
> git diff main feature/ログイン機能
📦 こんなとき：作業途中でブランチを切り替えないといけないとき
変更を一時退避する（stash）
使うシーン：作業中だがコミットするほどでもない状態でブランチを切り替えたいとき。変更を一時保存して、後で取り出せます。
変更を一時退避する
> git stash
退避リストを確認する
> git stash list
退避した変更を取り出す（最新のもの）
> git stash pop
↩️
元に戻す・やり直す
ミスしても大丈夫。Gitなら過去に戻れます。
↩️ こんなとき：ファイルの変更をコミット前に取り消したいとき
ファイルの変更を取り消す
使うシーン：編集した内容を「やっぱりやめよう」と思ったとき、最後のコミット時点の状態に戻します。
特定ファイルの変更を取り消す
> git restore index.html
ステージ（add）を取り消す（変更は残る）
> git restore --staged index.html
⚠️
git restore でファイルの変更を取り消すと、その変更は完全に消えます。元に戻せないので慎重に！
🔄 こんなとき：特定のコミットの変更を打ち消したいとき（履歴は残す）
コミットを安全に打ち消す（revert）
使うシーン：すでに push したコミットを「なかったことにしたい」とき。reset と違い、履歴を消さずに「打ち消すコミット」を新しく追加します。チームで使うときはこちらが安全。
手順1：コミットIDを確認する
> git log --oneline
手順2：確認したコミットIDを指定して打ち消す
> git revert abc1234
⏪ こんなとき：まだpushしていないコミットを取り消したいとき
コミットを取り消す（reset）
使うシーン：直前のコミットを「なかったことにしたい」ローカル限定の操作。--soft は変更を残し、--hard は変更ごと消します。
直前のコミットを取り消す（変更はステージに残る）
> git reset --soft HEAD~1
直前のコミットを取り消す（変更も消える ← 要注意）
> git reset --hard HEAD~1
⚠️
--hard は変更内容ごと削除されます。push 済みのコミットに使うと他のメンバーに影響が出るので、ローカルのみに留めましょう。push済みは revert を使って。

Quick Reference

チートシート（一覧表）

よく使うコマンドをまとめて確認。印刷して手元に置いておこう。

コマンド 何をするか こんなとき使う
git init リポジトリを初期化 新しいプロジェクトを Git で管理し始めるとき
git clone <URL> リモートをローカルにコピー GitHub のリポジトリを手元にダウンロードするとき
git status 現在の状態を確認 迷ったらまずこれ。今どのファイルが変更されているかわかる
git add <ファイル> ファイルをステージに追加 コミットに含めたいファイルを選ぶとき
git add . 変更ファイルをすべてステージ 全部まとめてコミットしたいとき
git commit -m "メッセージ" 変更を記録（セーブ） 作業の区切りで変更を保存するとき
git log --oneline コミット履歴を確認 過去の変更を振り返るとき、コミットIDを調べるとき
git diff 変更差分を表示 コミット前に何を変えたか確認するとき
git branch ブランチ一覧を表示 今どのブランチにいるか確認するとき
git checkout -b <名前> 新ブランチを作成して移動 新機能や修正の作業を始めるとき
git checkout <名前> ブランチを切り替える 別の作業ブランチに移動するとき
git merge <ブランチ名> ブランチを合流させる 作業ブランチを main に取り込むとき
git push ローカルをリモートにアップ コミットを GitHub に反映するとき
git pull リモートの変更を取り込む チームの最新状態を手元に反映するとき（毎日最初にやろう）
git stash 変更を一時退避 作業途中で別ブランチに移らないといけないとき
git restore <ファイル> ファイルの変更を取り消す 編集を全部やり直したいとき（注意: 戻せない）
git revert <コミットID> コミットを安全に打ち消す push済みのコミットを取り消したいとき（履歴は残る）
git reset --soft HEAD~1 直前のコミットを取り消す（変更は残る） ローカルのコミットをやり直したいとき
🚀
よく使う黄金フロー
git pull → 作業 → git add → git commit -m "..." → git push
毎日このサイクルを回すだけで Git の基本は習得できます！
