Section 01

コンフリクトとは？

コンフリクト（conflict）とは「競合・衝突」という意味。Gitでは同じファイルの同じ行を、複数の人が別々に変更してしまったときに発生します。

💥

何が起きるか

マージやプルをしたとき、Gitが「どちらの変更を採用すればいいかわからない」と判断すると、ファイル内にマーカー記号（特殊な印）を書き込んで処理を止めます。

🤔

なぜ起きるか

Gitは行単位で変更を追跡しています。別々のブランチでまったく同じ行を異なる内容に変えると、どちらが正しいかGitには判断できません。

🛠️

どう解決するか

コンフリクトはGitが自動で解決できないため、人間が手動でどちらの変更を使うか決めます。手順は決まっているので、落ち着いて対処できます。

💡
わかりやすいたとえ：
Google ドキュメントをオフライン中に2人が同時に同じ段落を書き換えた状態です。オンラインに戻ったとき「どちらの内容を使いますか？」と聞かれる状況と同じ。Gitはそれをファイル内のマーカーで示してくれます。

Section 02

コンフリクトが発生する状況（図解）

どういう状況でコンフリクトが起きるか、ブランチの流れで確認しましょう。

📝 main ブランチの変更（Aさん）

# index.html の5行目 <h1>Gitで学ぼう！</h1>

📝 feature ブランチの変更（Bさん）

# index.html の5行目（同じ場所！） <h1>Gitを一緒に学ぼう！</h1>
⚠️
main ブランチへのマージを試みると…
Gitが「同じ行が2通りに変更されている！どっちを使えばいいの？」と判断できず、CONFLICT（コンフリクト）エラーが発生します。ターミナルに CONFLICT (content): Merge conflict in index.html と表示されます。
$ git merge feature/ブランチ名 Auto-merging index.html CONFLICT (content): Merge conflict in index.html Automatic merge failed; fix conflicts and then commit the result. # → コンフリクトが発生したファイルを確認する $ git status

Section 03

コンフリクトマーカーの読み方（図解）

コンフリクトが発生すると、Gitはファイルの中に特殊な記号（マーカー）を書き込みます。この3種類のマーカーを理解すれば、何をどう直せばいいか分かります。

index.html CONFLICT
1<!DOCTYPE html>
2<html lang="ja">
3<body>
4
5<<<<<<< HEAD
6<h1>Gitで学ぼう！</h1>
7=======
8<h1>Gitを一緒に学ぼう！</h1>
9>>>>>>> feature/intro-update
10
11</body>
12</html>
◀ <<<<<<< HEAD
自分（現在のブランチ）の変更
HEADとは「今いるブランチの最新コミット」のこと。このマーカーから ======= の間が自分の変更内容です。
▶ =======
区切り線（セパレーター）
自分の変更と相手の変更を区切る境界線です。上が自分、下が相手の変更内容です。
▶ >>>>>>> ブランチ名
相手（マージしてきたブランチ）の変更
======= からこのマーカーまでが相手のブランチの変更内容。ブランチ名がここに表示されます。
💡
解決後のファイルイメージ（マーカーをすべて取り除いた状態）
<h1>Gitを一緒に学ぼう！</h1>　← どちらか（または両方を組み合わせた）内容だけ残す

Section 04

コンフリクトの解決手順（ステップバイステップ）

コンフリクトの解決は5つのステップで完了します。焦らず順番に進めましょう。

1

コンフリクトが発生したファイルを確認する

git status を実行してどのファイルでコンフリクトが起きているか確認します。「both modified」と表示されているファイルが対象です。

$ git status # 出力例： # On branch main # You have unmerged paths. # (fix conflicts and run "git commit") # # Unmerged paths: # both modified: index.html ← これがコンフリクトしているファイル
2

コンフリクトしているファイルを開く

VS Code などのエディタでファイルを開きます。<<<<<<< で始まる箇所がコンフリクト部分です。ファイル内を検索（Ctrl+F）して「<<<<<<<」を探すと素早く見つかります。

⚠️
マーカー（<<<<<<< ======= >>>>>>>）はGitが自動で書き込んだものです。このまま保存するとファイルが壊れるので、必ず削除してからコミットしてください。
3

どちらの変更を採用するか決めて編集する

3つのパターンから選びます。どれを選ぶかはチームで相談して決めましょう。

パターン A：自分の変更を採用
# マーカーと相手の変更を削除し、自分の変更だけ残す <h1>Gitで学ぼう！</h1>
パターン B：相手の変更を採用
# マーカーと自分の変更を削除し、相手の変更だけ残す <h1>Gitを一緒に学ぼう！</h1>
パターン C：両方を組み合わせる
# マーカーをすべて削除し、両方の良いところを組み合わせる <h1>Gitを一緒に楽しく学ぼう！</h1>
4

修正したファイルをステージに追加する

編集が終わったら git add で「コンフリクトを解決しました」とGitに伝えます。

# 解決したファイルをステージに追加 $ git add index.html # 複数のファイルをまとめて追加する場合 $ git add . # statusで確認（"both modified" が消えればOK） $ git status
5

コミットして完了！

すべてのコンフリクトを解決したらコミットします。メッセージには「コンフリクトを解決した」旨を書くと、後から見返したときに分かりやすいです。

$ git commit -m "コンフリクトを解決: index.htmlのh1タグ" # ログで確認 $ git log --oneline
🎉
コミットが完了すればマージ完了です！コンフリクトが怖くなくなってきましたか？

Section 05

VS Code / Cursor でコンフリクトを解決する

VS Code や Cursor にはコンフリクトを GUI で解決できる機能が組み込まれています。コマンドを打つより簡単に解決できます。

index.html CONFLICT
Accept: ✅ 現在の変更を使う（自分） ✅ 受信した変更を使う（相手） ✅ 両方の変更を使う 🔍 変更の比較
5 <<<<<<< HEAD　　　　　（現在の変更）
6 <h1>Gitで学ぼう！</h1>
7 =======
8 <h1>Gitを一緒に学ぼう！</h1>
9 >>>>>>> feature/intro-update　（受信した変更）
現在の変更を使う

自分のブランチ（HEAD）の内容を採用し、相手の変更を破棄します。

受信した変更を使う

マージしてきた相手ブランチの内容を採用し、自分の変更を破棄します。

両方の変更を使う

自分と相手の変更を両方残します（上に自分、下に相手）。その後手動で整形が必要な場合もあります。

💡
GUI で解決した後も git add と git commit は必要です！
ボタンをクリックしてマーカーが消えたら、ファイルを保存して git add ファイル名 → git commit を実行して解決完了にしましょう。

Section 06

コンフリクトを防ぐコツ

完全には防げませんが、頻度を大きく減らすことができます。チームで意識してみましょう。

🌅

毎日作業前に git pull

朝一番にリモートの最新状態を取り込むことで、自分のブランチを常に最新に保ちます。差分が小さいほどコンフリクトのリスクが下がります。

📦

小さく・こまめにコミット＆プッシュ

大きな変更を一度にまとめるより、小さな単位でコミット＆プッシュすると、他のメンバーが早く最新状態を把握でき、同じ箇所を触るリスクが減ります。

🗣️

「このファイル触ります」と声かけ

同じファイルを複数人が同時に編集するのが最大の原因です。Slack などで「〇〇を編集中です」と一声かけるだけでコンフリクトを大幅に防げます。

✂️

ブランチを短命に保つ

ブランチを長期間使い続けると、その間にmainが進んで差分が大きくなります。1機能・1修正 = 1ブランチを意識し、早めにPull Requestを出しましょう。

Quick Reference

コンフリクト解決 チートシート

いざというときにすぐ参照できるまとめです。

コマンド / 操作 何をするか タイミング
git status コンフリクトしているファイルを確認 マージ後に「CONFLICT」が出たらまずこれ
git diff コンフリクトの詳細を端末上で確認 ファイルを開く前に全体を把握したいとき
エディタでファイルを編集 マーカーを削除し、採用する内容だけ残す コンフリクト箇所を見つけたら手動編集
git add <ファイル> 解決済みをGitに伝える ファイルの編集・保存が終わったら
git commit マージ完了のコミットを作成 すべてのファイルをaddした後
git merge --abort マージを中断して元の状態に戻す 解決が難しく、いったんやり直したいとき
git checkout --ours <ファイル> 自分の変更だけを採用（コマンドで一括） 「完全に自分の版を使う」と決めたとき
git checkout --theirs <ファイル> 相手の変更だけを採用（コマンドで一括） 「完全に相手の版を使う」と決めたとき
🚀
コンフリクト解決の黄金フロー
git status → ファイルを開いて編集 → git add → git commit
このサイクルを覚えておけば、どんなコンフリクトでも落ち着いて対処できます！
